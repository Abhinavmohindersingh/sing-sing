System Overview
🗺️
What we're building
END-TO-END MARKETING AUTOMATION PIPELINE

A full-stack AI marketing platform that takes a user brief — whether a structured form, a plain English prompt, or an automated intake trigger — and generates platform-ready content (copy, images, video) via a network of specialised AI agents. Content flows through a human review layer before entering a managed publishing queue connected to live social APIs. All performance data feeds back into a RAG optimisation loop.

React Frontend Node / Python Backend PostgreSQL + pgvector AWS S3 Meta API LinkedIn API TikTok API

Layer 4 — Temporary Draft Storage
💾
Draft Layer — content_drafts table
PostgreSQL + AWS S3 FOR MEDIA

Every piece of generated content must be stored as a draft with the following schema before it enters the review dashboard:

-- content_drafts table schema
content_id    UUID         -- primary key
campus       VARCHAR     -- e.g. "brisbane"
course       VARCHAR     -- e.g. "carpentry"
platform     VARCHAR     -- instagram / linkedin / tiktok
agent_type   VARCHAR     -- content / image / video / campaign
status       VARCHAR     -- DRAFT / APPROVED / REJECTED / EDIT_REQUESTED / MANUALLY_EDITED
created_at   TIMESTAMP
media_url    TEXT        -- S3 URL for images / video
copy_text    TEXT        -- generated copy
edit_notes   TEXT        -- reviewer notes on edit requests
rejection_reason TEXT
Layer 5 — Marketing Team Review Dashboard
🖥️
Review Dashboard UI
CURRENTLY PARTIALLY IN DEMO — NEEDS FULL IMPLEMENTATION

The demo shows Approve / Edit / Reject buttons. The full review dashboard needs to be a proper page with a queue of all pending drafts, filterable by campus, course, platform, and agent type.

A

Approve HIGH

Sets status = APPROVED. Moves content to Publishing Queue. Triggers platform-specific formatting (aspect ratios, character limits, hashtag rules).

status → APPROVED → publish_queue
R

Reject CRITICAL

Sets status = REJECTED. Requires a mandatory rejection reason input. Archives the draft. Sends the rejection reason + original content as a negative embedding to the RAG optimisation layer.

status → REJECTED reason → RAG negative signal
E

Manual Edit HIGH

Team edits copy directly inline. Sets status = MANUALLY_EDITED. The diff between AI output and final edited version is logged and sent as a corrective signal to RAG.

status → MANUALLY_EDITED diff → RAG corrective signal
Q

Request Edit (from Agent) MEDIUM

Sets status = EDIT_REQUESTED. Edit notes attached. Re-routes back to the relevant sub-agent with the notes attached as context. Sub-agent generates a new draft.

status → EDIT_REQUESTED → re-routed to sub-agent
Layer 6 — Publishing Queue + Social API Integration
🚀
Publishing Queue
Redis / Celery TASK QUEUE + PostgreSQL publish_queue TABLE

Once approved, content enters a managed publishing queue. Use Redis + Celery (or BullMQ if staying in Node) for job scheduling. The queue handles rate limits, retry logic, and platform-specific formatting before calling social APIs.

  • ✦
    Facebook & Instagram — Meta Graph API. Use media_publish endpoint. Handle image upload as a two-step process (container → publish). OAuth2 with page token.
  • ✦
    LinkedIn — LinkedIn Marketing API. Use ugcPosts endpoint. Requires organisation URN and OAuth2 with w_member_social scope.
  • ✦
    TikTok — TikTok Content Posting API. Video only — use /video/upload endpoint. Requires TikTok for Business account.

Layer 8 — Memory & Learning Layer (RAG Optimisation)
🔁
RAG Optimisation Loop
pgvector — STORES PERFORMANCE EMBEDDINGS

This is the flywheel. Every signal from the review and analytics layers gets stored as a vector embedding and retrieved at generation time to bias future outputs. Three signal types:

✦ Corrective Signal
Edit history + diff stored as corrective learning signal. Tells the model: "human preferred this phrasing over what you wrote."
✦ Negative Signal
Rejected content + rejection reason stored as negative embedding. Prevents similar outputs from being generated again.
✦ Positive Signal
Approved content with high performance score stored as positive embedding. Biases future generations toward proven patterns.
✦ Implementation
Use pgvector's cosine similarity search. At generation time, retrieve top-K similar past jobs and include their outcomes in the LLM context window.
